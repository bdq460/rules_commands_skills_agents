## 用户访谈指南

### 访谈前准备

#### 1. 明确访谈目标

- 了解用户使用场景
- 识别用户痛点和需求
- 收集用户对现有系统的反馈
- 验证产品假设

#### 2. 准备访谈问题

**背景问题**：

- 您通常在什么情况下使用这个功能？
- 这个功能对您的工作有什么帮助？
- 您会如何描述这个功能的价值？

**痛点问题**：

- 使用这个功能时，您遇到的最大困难是什么？
- 哪些操作让您感到困惑或沮丧？
- 您是否需要手动完成某些步骤？
- 您是否需要重复某些操作？

**期望问题**：

- 您希望这个功能如何改进？
- 如果可以实现任何功能，您最想要什么？
- 有没有什么功能是您一直想要的但没有的？

**验证问题**：

- 如果我理解正确，您的主要需求是...对吗？
- 您提到的[痛点]是指...吗？
- 您认为[解决方案]能够解决您的问题吗？

#### 3. 准备访谈环境

- 安静的访谈场所
- 准备录音设备（经同意）
- 准备访谈记录表
- 准备产品原型或演示

### 访谈中

#### 开场（5分钟）

```text
感谢您抽出时间参与本次访谈。我想了解一下您对[产品/功能]的使用体验和需求。

访谈大约需要30-60分钟，我会询问一些关于您使用场景和需求的问题。

您的反馈对我们要重要，没有正确或错误的答案。请畅所欲言。

如果有任何您不想回答的问题，请随时告诉我。

今天的访谈内容会保密，只用于产品改进。

您有什么问题吗？

```

#### 主体访谈（30-50分钟）

1. **了解用户背景**（5分钟）
   - 您的角色和职责是什么？
   - 您的日常工作和任务是什么？
   - 您如何使用我们的产品？

2. **深入了解使用场景**（15-20分钟）
   - 您通常在什么情况下使用[功能]？
   - 您使用这个功能的频率如何？
   - 您能描述一次典型的使用场景吗？

3. **挖掘痛点和需求**（10-15分钟）
   - 使用这个功能时，您遇到的最大困难是什么？
   - 有没有什么步骤让您感到困惑或沮丧？
   - 您是否需要手动完成某些重复性的操作？
   - 如果可以实现任何改进，您最想要什么？

4. **验证和澄清**（5-10分钟）
   - 如果我理解正确，您的主要需求是...对吗？
   - 您能举个例子说明吗？
   - 您还有什么想补充的吗？

#### 结束（5分钟）

```text
非常感谢您的宝贵时间。您的反馈对我们非常有帮助。

如果您之后想起任何其他想法或需求，欢迎随时联系我们。

我们会将您的反馈整理成需求文档，并在后续与您确认。

再次感谢！

```

### 访谈后

#### 1. 整理访谈记录

- 回听录音（如果有）
- 整理关键信息
- 标记重要引用
- 识别共同模式

#### 2. 提取需求

- 从访谈记录中提取用户需求
- 区分功能需求和非功能需求
- 识别优先级
- 记录用户故事

#### 3. 验证理解

- 总结关键发现
- 与用户确认理解是否正确
- 询问是否有遗漏

## 痛点分析模板

### 痛点分析框架

#### 1. 痛点识别

**识别方法**：

- 用户访谈
- 用户反馈
- 数据分析
- 竞品分析
- 内部分析

**痛点分类**：

| 分类       | 说明               | 示例                 |
| ---------- | ------------------ | -------------------- |
| 功能缺失   | 需要的功能不存在   | 需要批量导出功能     |
| 功能不足   | 现有功能不够完善   | 搜索结果不够准确     |
| 性能问题   | 响应速度慢         | 页面加载时间过长     |
| 易用性问题 | 操作复杂或不直观   | 按钮位置不明显       |
| 兼容性问题 | 在特定环境下不工作 | 在IE浏览器中显示异常 |

#### 2. 痛点评估

**评估维度**：

| 维度     | 说明                             | 评分 |
| -------- | -------------------------------- | ---- |
| 影响程度 | 对用户工作的影响（高/中/低）     |      |
| 发生频率 | 痛点发生的频率（频繁/偶尔/少见） |      |
| 影响范围 | 受影响的用户数量                 |      |
| 业务价值 | 解决痛点带来的业务价值           |      |
| 实现难度 | 技术实现难度（高/中/低）         |      |

**优先级计算**：

优先级 = (影响程度权重 × 影响程度分数) +
(发生频率权重 × 发生频率分数) +
(业务价值权重 × 业务价值分数) -
(实现难度权重 × 实现难度分数)

#### 3. 痛点分析表

| 序号 | 痛点描述 | 分类 | 影响程度 | 发生频率 | 影响范围 | 业务价值 | 实现难度 | 优先级 |
| ---- | -------- | ---- | -------- | -------- | -------- | -------- | -------- | ------ |
| 1    |          |      |          |          |          |          |          |        |
| 2    |          |      |          |          |          |          |          |        |

### 痛点分析示例

#### 案例：报表导出功能

**痛点识别**：

| 序号 | 痛点描述                            | 分类     | 来源     |
| ---- | ----------------------------------- | -------- | -------- |
| 1    | 只能单个导出报表，无法批量导出      | 功能缺失 | 用户访谈 |
| 2    | 导出等待时间长，大文件导出超过5分钟 | 性能问题 | 用户反馈 |
| 3    | 导出的Excel格式不符合财务部门要求   | 功能不足 | 内部分析 |

**痛点评估**：

| 序号 | 痛点           | 影响程度 | 发生频率 | 影响范围 | 业务价值 | 实现难度 | 优先级 |
| ---- | -------------- | -------- | -------- | -------- | -------- | -------- | ------ |
| 1    | 单个导出       | 高       | 频繁     | 50人     | 高       | 中       | P0     |
| 2    | 导出速度慢     | 中       | 频繁     | 30人     | 中       | 高       | P1     |
| 3    | 格式不符合要求 | 高       | 频繁     | 50人     | 高       | 低       | P0     |

**分析结论**：

- P0优先级痛点（2个）：
  - 批量导出功能
  - 导出格式适配

- P1优先级痛点（1个）：
  - 导出性能优化

### 用户故事示例

#### 示例1：批量导出功能

```text
作为财务人员，
我想要批量导出多个报表，
以便提高月底汇总效率。

### 验收标准

- Given 用户在报表列表页，
  When 用户勾选多个报表并点击批量导出，
  Then 系统开始生成所有选中的报表

- Given 所有报表生成完成，
  When 用户点击下载，
  Then 下载包含所有报表的ZIP文件

- Given 用户勾选的报表数量超过系统限制，
  When 用户点击批量导出，
  Then 系统提示"最多只能选择10个报表"

```

## 需求优先级评估

### 评估方法

#### RICE评分法

```Math
    RICE = (Reach × Impact × Confidence) / Effort
```

- **Reach (覆盖范围)**：有多少用户会受到影响（人数）
- **Impact (影响力)**：对每个用户的影响程度（0.25-3分）
- **Confidence (信心度)**：对评估结果的信心（0-100%）
- **Effort (工作量)**：实现所需的工作量（人月）

#### WSJF（加权最短作业优先）

```Math
    WSJF = (商业价值 + 时间紧迫性 + 风险降低/机会促成) / 工作量
```

### 优先级矩阵

| 优先级 | 商业价值 | 紧急度 | 复杂度 | 说明           |
| ------ | -------- | ------ | ------ | -------------- |
| P0     | 高       | 紧急   | 低     | 必须立即完成   |
| P1     | 高       | 紧急   | 中/高  | 应该尽快完成   |
| P2     | 中       | 一般   | 中     | 可以计划完成   |
| P3     | 低       | 不急   | 高     | 可以延后或不做 |

## 最佳实践

### 1. 需求收集

- 从用户视角描述需求
- 聚焦业务价值而非技术实现
- 收集真实场景而非假设
- 保持开放心态，不预设答案

### 2. 用户访谈

- 积极倾听，不打断用户
- 问开放式问题，避免引导性提问
- 要求用户提供具体例子
- 验证理解是否正确

### 3. 痛点分析

- 区分症状和根因
- 收集数据支持分析
- 考虑多个用户群体的差异
- 平衡技术可行性和业务价值

### 4. 需求文档

- 使用清晰、简洁的语言
- 避免技术术语和抽象概念
- 提供具体示例和场景
- 保持文档更新
